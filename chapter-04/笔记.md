## 本章目标

1. 编写**可复用、可扩展、跨网站**的通用爬虫思路  
2. 理解 **数据模型 + 配置分离**：解析规则不进业务硬编码  
3. 使用 **`Website` + `Crawler` + `Content`** 三层分工  
4. **一套解析流程**适配多站：改配置为主、改引擎为辅  
5. 避免「一站一脚本」导致的重复与分叉维护  

---

## 1. 核心知识点

### 1.1 什么是「爬虫模型」？

把爬虫拆成三块：

| 部分 | 职责 |
|------|------|
| **Content（数据模型）** | 统一存放结果：URL、标题、正文等 |
| **Website（配置模型）** | 每站的 **CSS 选择器**、名称、根 URL 等「抓取指令」 |
| **Crawler（引擎）** | 请求、按 `Website` 抽取、组装为 `Content` |

### 1.2 三大核心类

1. **Content**  
   - 各站最终都落到**同一字段集合**，便于入库、对比、分析。  

2. **Website**  
   - 只描述「**去哪取**」：如标题在 `h1`、正文在 `div.content p`。  
   - 改版时优先改**配置**，而不是复制整段爬虫逻辑。  

3. **Crawler**  
   - 统一 `get_page` → `safe_get` → `parse`。  
   - 可选 **`parse_from_soup`**：对已拿到的 `BeautifulSoup` 做纯解析（离线 fixture、单测）。  

### 1.3 安全提取：`safe_get`

- `soup` 为 `None` 或选择器无匹配时返回 **`""`**，不抛异常。  
- 多节点时用 **`"\n".join(...)`** 合并段落。  
- 与第 2 章「先判断再 `.`」一致，是稳健爬虫的标配。  

### 1.4 CSS 选择器

- `soup.select("h1")`、`".class"`、`"#id"`、后代组合等。  
- 相对 `find`/`find_all`，选择器表达力更强、配置更短。  

### 1.5 请求与限速

- 使用 **`requests.Session()`** 复用连接、统一 **User-Agent**。  
- 在引擎层加 **`time.sleep(delay_sec)`**（见 `crawler_model.py`），在**成功拿到响应后**再休眠，避免失败请求也拖慢重试。  
- **`get_page`** 对 **`RequestException`** 做 **最多 3 次**短间隔重试，缓解偶发断连。

## 2. 易错点

1. **配置与代码搅在一起** → 换站就要动引擎，易回归错误。  
2. **不做安全抽取** → 选择器失效即 `AttributeError` / 空迭代。  
3. **各站字段不一致** → 下游无法统一分析；应通过 **Content** 对齐。  
4. **`parse(site, url)` 传错参数** → 第二个参数必须是**当前要抓的页面 URL**，不是把整个站点列表当 URL 传入。  
5. **线上示例选择器长期不变** → 新闻站改版快，书中 Reuters / NYTimes 等 **class 可能已变**，以开发者工具为准。

## 4. 本章总结

1. **模型 = 数据（Content）+ 配置（Website）+ 引擎（Crawler）**  
2. **多站扩展**主要靠增加/修改 `Website` 实例  
3. **`safe_get`** 降低因 DOM 微变导致的崩溃  
4. **CSS 选择器**适合作为「可序列化配置」的载体  
5. **统一 Content** 便于存储、统计与后续管道（数据库、队列等）
